## 1. MongoDB (NoSQL 대표)
**"유연함과 확장성의 대명사"**
데이터를 엑셀 표(Table)가 아닌 **JSON 문서(Document)** 형태로 저장하는 방식입니다.

### 장점
* **Schema-less (유연성)**: 테이블 구조를 미리 정의할 필요가 없습니다. 필드가 계속 추가되거나 구조가 자주 변하는 데이터(게임 로그, 비정형 데이터) 처리에 최적입니다.
* **Scale-out (수평 확장)**: 데이터가 많아지면 서버를 옆으로 늘리는 '샤딩(Sharding)' 기능이 강력하여 대용량 데이터 처리가 매우 쉽습니다.
* **개발 편의성**: 객체 지향 언어(Java, JS, Python)의 객체(Object)를 변환 없이 그대로 DB에 넣고 뺄 수 있어 개발 생산성이 높습니다.
* **고속 쓰기**: 복잡한 제약 조건 검사가 적어, 단순 쓰기 작업에서 압도적인 성능을 보입니다.

### 단점 
* **JOIN의 부재**: 관계형 DB처럼 테이블을 연결하는 기능이 매우 약합니다. 따라서 데이터 중복 저장이 불가피하며, 데이터 수정 시 관리 비용이 듭니다.
* **데이터 일관성 비용**: 트랜잭션(ACID)을 지원하지만, RDBMS만큼 견고하거나 빠르지 않아 금융/결제 시스템에는 신중해야 합니다.
* **메모리 사용량**: 성능을 위해 메모리(RAM)를 많이 사용하므로 하드웨어 비용이 높을 수 있습니다.

## 2. 데이터베이스 상세 비교표

| 비교 항목 | **MongoDB** (NoSQL) | **MySQL** (RDBMS) | **PostgreSQL** (ORDBMS) |
| :--- | :--- | :--- | :--- |
| **저장 방식** | **JSON 문서 (Document)** | **테이블 (Table / Row)** | **테이블 + 객체 (Row & Object)** |
| **스키마 (구조)** | **없음 (자유로움)** | **엄격함 (변경 어려움)** | 엄격함 + 유연함 (JSONB 지원) |
| **확장성** | **수평 확장 (Scale-out) 쉬움** | 수직 확장 (Scale-up) 위주 | 수직 확장 위주 |
| **JOIN (관계)** | 비효율적 / 지원 미비 | **매우 강력하고 빠름** | **매우 강력하고 빠름** |
| **트랜잭션** | 지원하지만 성능 저하 있음 | **완벽 지원 (표준)** | **완벽 지원 (매우 강력)** |
| **주요 용도** | 로그, 채팅, 실시간 빅데이터 | **회원, 결제, 아이템 (범용)** | **복잡한 통계, 지도(GIS), 분석** |

## 3. MySQL vs PostgreSQL: RDBMS 형제의 차이


> **"심플하고 빠르며 가장 대중적임"**
* **장점**: 
    * 단순 **읽기(Read)** 성능이 매우 뛰어납니다.
    * 설정과 관리가 비교적 쉽고, 전 세계적으로 레퍼런스가 가장 많습니다.
    * 업데이트가 빈번한 일반 웹 서비스나 게임 서버에 적합합니다.
* **단점**: 
    * 복잡한 쿼리나 대용량 통계 처리에서는 PostgreSQL보다 성능이 떨어질 수 있습니다.

### 🐘 PostgreSQL
> **"기능이 풍부하고 학구적인 완벽주의자"**
* **장점**: 
    * **복잡한 쿼리(Complex Query)**와 대용량 데이터 분석 처리에 강합니다.
    * **JSON 지원(JSONB)**이 매우 강력해서, MongoDB의 장점(NoSQL적 특성)도 일부 흡수했습니다.
    * **PostGIS** 확장 기능을 통해 위치 기반(지도) 서비스에서 독보적입니다.
* **단점**: 
    * MySQL보다 설정이 까다롭고 무거운 편입니다.
    * 단순 조회 속도는 MySQL이 근소하게 앞서는 경우가 많습니다.
